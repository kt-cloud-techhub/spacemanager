# 워크플로우 규칙

## 적응형 워크플로우 원칙

워크플로우는 작업에 맞게 적응합니다. 반대가 아닙니다.

다음을 기반으로 필요한 단계를 평가합니다:
1. 사용자의 의도와 명확성
2. 기존 코드베이스 상태
3. 복잡성과 범위
4. 위험 및 영향도

---

## 확장 규칙 적용

- 확장 규칙은 선택적 지침이 아닌 하드 제약입니다.
- 각 단계에서 해당 단계의 맥락에 적용 가능한 규칙만 적용합니다.
- 적용 불가한 규칙 → 준수 요약에 N/A로 표시합니다.
- 적용 가능한 규칙을 위반하면 **블로킹 발견사항**입니다.
- 적용 전 `aidlc-docs/aidlc-state.md` → `## Extension Configuration`에서 활성화/비활성화 상태를 확인합니다.
- 기본값: 설정이 없으면 적용. `## Applicability Question`이 없는 확장은 항상 적용.

---

## 계획 체크박스 적용 규칙

1. 체크박스를 업데이트하지 않고 작업을 완료하지 않습니다.
2. 계획 파일의 단계를 완료한 즉시 해당 단계를 `[x]`로 표시합니다.
3. 작업이 완료된 동일한 상호작용에서 이루어져야 합니다.
4. 예외 없음: 모든 계획 단계 완료는 추적되어야 합니다.

두 가지 추적 수준:
- **계획 수준**: 각 단계 내 상세 실행 진행 상황.
- **단계 수준**: `aidlc-state.md`의 전체 워크플로우 진행 상황.

## 착수 단계 (INCEPTION PHASE)

**목적**: 계획, 요구사항 수집, 아키텍처 결정.
**초점**: 무엇을 왜 만들지 결정.

### 환경 분석 (항상 실행)
1. `audit.md`에 초기 사용자 요청을 완전한 원문으로 기록합니다.
2. `inception/workspace-detection.md`에서 단계를 로드합니다.
3. 기존 `aidlc-state.md` 확인 (있으면 재개). 워크스페이스 스캔. 브라운필드/그린필드 판단.
4. 다음 단계 결정: 기존 코드 분석 (브라운필드, 산출물 없음) 또는 요구사항 분석.
5. `audit.md`에 결과 기록. 완료 메시지 표시. 자동으로 다음 단계 진행.

### 기존 코드 분석 (조건부 — 브라운필드만)
실행 조건: 기존 코드베이스 감지 AND 이전 분석 산출물 없음.
건너뛰기: 그린필드 OR 산출물 이미 존재.

1. `audit.md`에 시작 기록.
2. `inception/reverse-engineering.md`에서 단계 로드.
3. 패키지, 컴포넌트, 비즈니스 트랜잭션, 아키텍처, API, 기술 스택, 의존성 분석.
4. **명시적 승인 대기** 후 진행. 사용자 응답 기록.

### 요구사항 분석 (항상 실행 — 적응형 깊이)
깊이: 최소 (단순 요청) / 표준 (일반) / 종합 (복잡, 고위험).

1. `audit.md`에 사용자 입력 기록.
2. `inception/requirements-analysis.md`에서 단계 로드.
3. 의도 분석, 깊이 결정, 요구사항 평가, 필요 시 명확화 질문.
4. **명시적 승인 대기** 후 진행. 사용자 응답 기록.

### 사용자 시나리오 (항상 실행)
한 단계 내 두 파트:
- **파트 1 — 계획**: 시나리오 계획 + 질문 → 답변 수집 → 모호성 분석 → 승인.
- **파트 2 — 생성**: 승인된 계획을 실행하여 시나리오와 페르소나 생성.

1. `audit.md`에 사용자 입력 기록.
2. `inception/user-stories.md`에서 단계 로드.
3. 지능형 평가 수행 (user-stories.md Step 1).
4. 파트 1 실행 후 파트 2 실행.
5. **명시적 승인 대기** 후 진행. 사용자 응답 기록.

### 실행 계획 수립 (항상 실행)
1. `audit.md`에 사용자 입력 기록.
2. `inception/workflow-planning.md`와 `common/content-validation.md`에서 단계 로드.
3. 이전 모든 맥락 로드 (기존 코드 분석, 의도, 요구사항, 사용자 시나리오).
4. 실행할 단계와 깊이 결정.
5. NFR 건너뛸 경우, `tech-stack-defaults.md`에서 자동 선택된 기술 스택을 표로 보여주고 사용자에게 확인.
6. 파일 생성 전 모든 콘텐츠 검증.
7. **명시적 승인 대기** — 사용자가 추천을 변경할 수 있음을 강조. 사용자 응답 기록.

### 애플리케이션 설계 (조건부 — 사용자 확인)

| 아키텍처 패턴 | 예시 | 설계 필요 여부 |
|---|---|---|
| 웹 애플리케이션 (Full-Stack) | 서비스 데스크, 게시판, 쇼핑몰 | **필수** — API/DB/화면 설계 |
| API 서비스 (API + DB) | REST API 백엔드, 마이크로서비스 | **필수** — API/DB 설계 |
| CLI / TUI | 명령줄 도구, 터미널 UI | 건너뛰기 가능 |
| 라이브러리 / SDK | npm 패키지, Python 라이브러리 | 건너뛰기 가능 |
| Slack Bot / Discord Bot | 메시징 플랫폼 자동화 | 건너뛰기 가능 |

결정 방법: AI가 위 기준에 따라 추천하되, AskUserQuestion 팝업으로 사용자에게 확인합니다.
최종 결정은 항상 사용자가 합니다 — AI가 자의적으로 건너뛰지 않습니다.

1. `audit.md`에 사용자 입력 기록.
2. `inception/application-design.md`에서 단계 로드.
3. 적절한 깊이로 실행.
4. **명시적 승인 대기** 후 진행. 사용자 응답 기록.

### 작업 단위 나누기 (조건부)
실행 조건: 여러 작업 단위로 분해 필요 / 여러 서비스 또는 모듈 필요.
건너뛰기: 단일 단순 단위 / 단일 컴포넌트 구현.

1. `audit.md`에 사용자 입력 기록.
2. `inception/units-generation.md`에서 단계 로드.
3. 적절한 깊이로 실행.
4. **명시적 승인 대기** 후 진행. 사용자 응답 기록.

## 구축 단계 (CONSTRUCTION PHASE)

**목적**: 상세 설계, NFR 구현, 코드 생성.
**초점**: 어떻게 만들지 결정.

각 단위는 다음 단위로 넘어가기 전에 완전히 완료됩니다 (설계 + 코드).

### 상세 기능 설계 (조건부, 단위별)
실행 조건: 새 데이터 모델, 복잡한 비즈니스 로직, 또는 비즈니스 규칙의 상세 설계 필요.
건너뛰기: 단순 로직 변경, 새 비즈니스 로직 없음.

1. `audit.md`에 사용자 입력 기록.
2. `construction/functional-design.md`에서 단계 로드.
3. 이 단위의 기능 설계 실행.
4. 표준화된 **2가지 선택지** 완료 메시지 표시 (functional-design.md에 정의된 대로). **3가지 선택지 등 임의 변형 금지.**
5. **명시적 승인 대기**. 사용자 응답 기록.

### 기술 스택 및 품질 요구사항 (조건부, 단위별)
실행 조건: 성능 요구사항, 보안 고려사항, 확장성 우려, 또는 기술 스택 선택 필요.
건너뛰기: NFR 요구사항 없음 또는 기술 스택 이미 결정됨.

1. `audit.md`에 사용자 입력 기록.
2. `construction/nfr-requirements.md`에서 단계 로드.
3. NFR 평가 실행.
4. 표준화된 **2가지 선택지** 완료 메시지 표시. **임의 변형 금지.**
5. **명시적 승인 대기**. 사용자 응답 기록.

### 성능·보안 설계 (조건부, 단위별)
실행 조건: 기술 스택 및 품질 요구사항 단계가 실행된 경우.
건너뛰기: 기술 스택 및 품질 요구사항 단계가 건너뛰어진 경우.

1. `audit.md`에 사용자 입력 기록.
2. `construction/nfr-design.md`에서 단계 로드.
3. NFR 설계 실행.
4. 표준화된 **2가지 선택지** 완료 메시지 표시. **임의 변형 금지.**
5. **명시적 승인 대기**. 사용자 응답 기록.

### 배포 환경 설계 (조건부, 단위별)
실행 조건: 인프라 서비스 매핑 필요, 배포 아키텍처 필요, 클라우드 리소스 명세 필요.
건너뛰기: 인프라 변경 없음, 이미 정의됨, 또는 로컬 개발만.

1. `audit.md`에 사용자 입력 기록.
2. `construction/infrastructure-design.md`에서 단계 로드.
3. 인프라 설계 실행.
4. 표준화된 **2가지 선택지** 완료 메시지 표시. **임의 변형 금지.**
5. **명시적 승인 대기**. 사용자 응답 기록.

### 코드 생성 (항상 실행, 단위별)
한 단계 내 두 파트:
- **파트 1 — 계획**: 체크박스가 있는 상세 코드 생성 계획 → 승인.
- **파트 2 — 생성**: 승인된 계획을 실행하여 코드, 테스트, 산출물 생성.

1. `audit.md`에 사용자 입력 기록.
2. `construction/code-generation.md`에서 단계 로드.
3. 파트 1 실행 후 파트 2 실행.
4. 표준화된 **2가지 선택지** 완료 메시지 표시. **임의 변형 금지.**
5. **명시적 승인 대기**. 사용자 응답 기록.

### 빌드 & 테스트 (항상 실행 — 모든 단위 완료 후)
1. `audit.md`에 사용자 입력 기록.
2. `construction/build-and-test.md`에서 단계 로드.
3. 생성: 빌드 지침, 단위 테스트 지침, 통합 테스트 지침, 성능 테스트 지침 (해당 시).
4. `build-and-test/` 하위 디렉토리에 파일 생성.
5. "빌드 및 테스트 지침이 완료되었습니다. Operations 단계로 진행할까요?" — **명시적 승인 대기**. 사용자 응답 기록.

## 운영 단계 (OPERATIONS PHASE)

향후 배포 및 모니터링 워크플로우를 위한 플레이스홀더입니다.
현재 상태: 모든 빌드 및 테스트 활동은 구축 단계에서 처리됩니다.

---

## 핵심 원칙

- **적응형 실행**: 가치를 더하는 단계만 실행합니다.
- **투명한 계획**: 시작 전 항상 실행 계획을 보여줍니다.
- **사용자 제어**: 사용자가 단계 포함/제외를 요청할 수 있습니다.
- **진행 추적**: `aidlc-state.md`에 실행 및 건너뛴 단계를 업데이트합니다.
- **완전한 감사 추적**: `audit.md`에 모든 사용자 입력과 AI 응답을 타임스탬프와 함께 기록합니다. 사용자 입력을 요약하지 않습니다.
- **품질 집중**: 복잡한 변경은 전체 처리, 단순한 변경은 효율적으로 유지합니다.
- **콘텐츠 검증**: 파일 생성 전 항상 `content-validation.md` 규칙에 따라 콘텐츠를 검증합니다.
- **임의 변형 금지**: 구축 단계는 반드시 표준화된 2가지 선택지 완료 메시지를 사용합니다. 3가지 선택지 메뉴 등 임의 변형을 만들지 않습니다.
