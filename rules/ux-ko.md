# UX 규칙

## 언어

- 모든 사용자 대면 메시지 (환영, 질문, 승인, 완료): **한국어**
- AI 내부 지침 및 규칙 파일 참조: 영어 유지
- 외래어 표기: 로컬(local), 서버(server), 데이터베이스(database), 컨테이너(container), 쿠버네티스(Kubernetes)

---

## 질문 응답 방식

- 객관식 → AskUserQuestion 팝업
- 주관식 → 채팅 직접 입력
- 질문 `.md` 파일 생성 금지 — 답변은 `aidlc-docs/audit.md`에 기록

---

## 단계 완료 형식

**중요**: 사용자에게 파일을 열어보라고 하지 마세요. 대신:

1. **핵심 내용을 채팅에 표시** — 표, 목록, 섹션으로 구조화합니다.
2. **파일은 백그라운드에서 저장** — 전체 문서는 `aidlc-docs/`에 저장. "파일 확인하세요"라고 안내하지 않습니다.
3. **AskUserQuestion 팝업으로 승인 요청** — 내용을 보여준 직후 팝업으로 승인/변경을 받습니다.

**브리핑 원칙**: 각 단계 완료 후 사용자는 파일을 읽지 않아도 채팅만으로 맥락을 파악할 수 있어야 합니다.

- 숫자 요약이 아닌 실제 내용을 보여줍니다 ("3개 페르소나, 9개 스토리"는 불충분).
- 각 항목의 제목, 핵심 설명, 구조를 보여줍니다.
- 상세 내용은 파일에 있지만, 채팅만으로도 충분히 이해 가능해야 합니다.

**완료 메시지 형식** (모든 단계 공통):

```markdown
# [단계명] 완료

[핵심 산출물 내용 — 표, 목록, 카드형 구조로 표시 — 줄글 금지]

---
다음은 **[다음 단계명]** 단계입니다.
[다음 단계에서 무엇을 하는지 1~2문장으로 설명]
```

그 직후 AskUserQuestion 팝업:

```yaml
AskUserQuestion:
  question: "위 [단계명] 결과를 승인하고 [다음 단계명]으로 진행할까요?"
  header: "승인"
  options:
    - label: "승인"
      description: "[다음 단계명]으로 진행합니다"
    - label: "변경 요청"
      description: "수정할 내용을 채팅으로 알려주세요"
```

**산출물 표시 형식**:
- API 목록 → 표 (메서드 | 경로 | 설명)
- DB 테이블 → 표 (컬럼명 | 타입 | 설명 | 제약조건)
- 기능 목록 → 번호 목록 + 간결한 설명
- 사용자 시나리오 → 카드형 구조 (제목, 설명, 완료 조건)
- 설계 결과 → 섹션별 (백엔드 / 프론트엔드 / DB) 표 또는 목록
- 긴 텍스트 블록 금지 — 반드시 구조화

**금지 사항**:
- ❌ "파일을 확인해주세요: `aidlc-docs/...`"
- ❌ "산출물을 검토해주세요"
- ❌ 줄글로 내용을 장황하게 나열
- ✅ 표, 목록, 섹션으로 구조화
- ✅ AskUserQuestion 팝업으로 승인/변경 받기
- ✅ 다음 단계가 뭔지, 거기서 뭘 하는지 안내

## 진행률 표시

매 단계 완료 시 완료 메시지 **상단**에 전체 진행 상태를 한 줄로 표시합니다.

형식 (확정된 실행 계획의 단계만 포함):

```
착수: ✅ 환경분석 → ✅ 요구사항 → ✅ 시나리오 → ✅ 실행계획 → ⬜ 설계
구축: ⬜ 코드 생성 → ⬜ 빌드 & 테스트
```

범례:
- ✅ = 완료
- 🔵 = 현재 진행 중
- ⬜ = 아직 안 함
- 건너뛴 단계는 표시하지 않음

예시 — 코드 생성 중:

```
착수: ✅ 환경분석 → ✅ 요구사항 → ✅ 시나리오 → ✅ 실행계획 → ✅ 설계
구축: 🔵 코드 생성 → ⬜ 빌드 & 테스트
```

예시 — /dev:improve 사이클:

```
개선: ✅ 분석 → 🔵 스펙 업데이트 → ⬜ 코드 수정 → ⬜ 테스트
```

| Workspace Detection | 환경 분석 |
| Reverse Engineering | 기존 코드 분석 |
| Requirements Analysis | 요구사항 분석 |
| User Stories | 사용자 시나리오 |
| Workflow Planning | 실행 계획 수립 |
| Application Design | 애플리케이션 설계 |
| Units Generation | 작업 단위 나누기 |
| Functional Design | 상세 기능 설계 |
| NFR Requirements | 기술 스택 및 품질 요구사항 |
| NFR Design | 성능·보안 설계 |
| Infrastructure Design | 배포 환경 설계 |
| Code Generation | 코드 생성 |
| Build and Test | 빌드 & 테스트 |

### 일반 기술 용어 변환

| 기술 용어 | 쉬운 표현 |
|---|---|
| NFR (Non-Functional Requirements) | 기술 스택 및 품질 요구사항 |
| Acceptance Criteria | 완료 조건 |
| Persona | 사용자 유형 |
| Domain Model / Entity | 데이터 구조 |
| Business Rule | 처리 규칙 |
| Endpoint / API | 기능 주소 (API) |
| Schema | 테이블 구조 |
| Unit of Work | 작업 단위 |
| Brownfield / Greenfield | 기존 프로젝트 / 새 프로젝트 |
| INCEPTION / CONSTRUCTION / OPERATIONS | 착수 / 구축 / 운영 |

## 실행 계획 완료 형식

```markdown
# 실행 계획 완료

다음 순서로 진행합니다:

1. **[단계]** — [한 줄 설명]
2. **[단계]** — [한 줄 설명]
```

그 직후 AskUserQuestion으로 선택적 단계 포함 여부를 질문합니다.
AI는 건너뛰는 단계를 일방적으로 통보하지 않습니다 — 항상 질문합니다: "추가로 진행할 수 있는 단계가 있습니다. 포함하시겠습니까?"

선택적 단계 질문 예시:

```yaml
question: "추가로 진행할 수 있는 단계가 있습니다. 포함하시겠습니까?"
header: "추가 단계"
options:
  - label: "바로 진행"
    description: "위 계획대로 진행합니다"
  - label: "상세 기능 설계 추가"
    description: "비즈니스 로직을 더 구체적으로 설계합니다"
  - label: "품질 요구사항 추가"
    description: "성능/보안 목표를 상세하게 정의합니다"
multiSelect: true
```

**실행 계획 출력 금지 사항**:
- ❌ Mermaid 다이어그램이나 복잡한 플로우차트
- ❌ 내부 용어: "EXECUTE", "SKIP", "CONDITIONAL"
- ❌ NFR, Units Generation 등 기술 스테이지명 그대로 노출
- ❌ 실행/건너뛰기 근거를 길게 설명
- ❌ AI가 "건너뛰는 단계: ..." 형식으로 일방적 통보
- ✅ 간단한 번호 목록으로 "진행 순서"
- ✅ 선택적 단계는 AskUserQuestion으로 질문
- ✅ 각 단계가 무엇을 하는지 한 줄 설명 포함

## 보안 확장 표시 방식

**중요**: 보안 준수 요약을 사용자에게 직접 보여주지 마세요.

- ❌ SECURITY-01 ~ SECURITY-15 규칙 ID를 나열하지 않음
- ❌ 완료 메시지에 "Security Compliance" 섹션 포함하지 않음
- ✅ 보안 규칙은 내부적으로 검증하되, 사용자에게는 결과만 간단히 표시
- ✅ 문제가 있을 때만 "보안 관련 수정이 필요합니다: [구체적 내용]"으로 알림
- ✅ 문제가 없으면 별도 언급 없이 진행

## 산출물 상세 수준

산출물은 실제 개발이 가능한 수준으로 구체적이어야 합니다. 너무 추상적이면 코드 생성 품질이 떨어집니다.

**요구사항 분석**:
- 기능 표: 기능명 | 설명 | 우선순위
- 비기능 요구사항에 구체적 수치 포함 (예: "10명 동시 접속", "1초 이내 응답")
- 제약 조건 명시 (예: "DB 서버 없이 파일 기반", "단일 컨테이너")

**사용자 시나리오**:

```markdown
### 시나리오 N: [제목]
**사용자**: [사용자 유형]
**상황**: [언제 이 상황이 발생하는지]
**행동**: [사용자가 하는 행동]
**결과**: [결과로 무슨 일이 일어나는지]

**완료 조건**:
- [ ] [조건 1]
- [ ] [조건 2]
```

- 최소 3~5개 시나리오 포함
- 각 시나리오에 완료 조건 반드시 포함
- 사용자 유형(페르소나)을 구체적으로 정의: 이름, 역할, 주요 행동

**애플리케이션 설계**:

API 표:
```
| 메서드 | 경로 | 설명 | 요청 바디 | 응답 |
```

DB 표:
```
| 컬럼 | 타입 | 설명 | 제약조건 |
```

화면 구성: 각 화면별 주요 구성 요소 설명 (필터, 목록, 버튼, 폼).

## 환영 메시지

**중요**: 소프트웨어 개발 요청 시작 시 `common/welcome-message-workshop.md`에서 환영 메시지를 로드하여 표시합니다.
새 워크플로우 시작 시 한 번만 표시합니다. 이후 상호작용에서는 다시 로드하지 않습니다.
